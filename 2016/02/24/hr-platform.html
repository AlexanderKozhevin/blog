HR-platform

HR - platform is a quite simple tool to organize & manage contacts, job applications and events. The idea was suggested from a friend of mine back in 2014. But only 1.5 years later it was completed. We just delayed a thing. But 6 months ago when I quit a regular job I decided to take matters into own hands. The development process itself wasn't really straight. I played with technologies and libraries. When I got the first version of it I decided to rewrite it completely. Cause it wasn't really friendly. But finally in two months I made it about the same I wanted it. Right now it's in closed beta.
And there are few interesting things I want to share with you.
Interface in 1 Minute
Interface itself is based on material design. I guess, it's a great solution for small projects and startups. Cause minus one huge headache is taken away.

Angular Material is used within this project. It has number of primary elements and very probably it's more than enough. Generally it feels like a medium quality lego-constructor. Excellent for small apps, so-so for average and difficult for large ones.
Front-end architecture
Application is based on Angular 1.4.8, Angular-Material. Thanx to Andrey Yamanov I decided to use component approach in conjunction with default one - when all the files are split by type and view logic. It helped me to isolate elements with really complex logic. The same way I act in controllers. I try to hide all the routine code in factories and concentrate all the logic in controllers as much as it possible. Even charts configurations are moved to factories.

Project builder is Gulp. And pretty much syntax sugar is used - Jade, Coffee-Script, SCSS. To simplify browser extensions support I use bourbon, a framework not a drink.

Back-end architecture

I guess Sails.js is my favorite node.js framework. It's easy to use, easy to customize. And it has a number of 3rd party things which makes it even better. Like - Waterline ORM. In conjunction with Restangular it makes server communication just effortless. Currently MongoDB for common data and Redis as a session storage are used.
I prefer to write methods in Coffee-Script since sails has build-in support.

For images storage I use S3Amazon for which sails has adapter as also!

Our platform has integrated statistics. Currently it collects page visits data. But technically we gather almost any kind of data since we use Mixpanel. Front-end application has angular Mixpanel library which allows to send statistics about events, clicks, orders etc. And we can easily select which ones we would like to have and which are not. Of course statistics is available only in admin panel via server api. Last one communicate with Mixpanel service itself.

I installed and configured NGINX to be able to serve two apps, which attached to domain and subdomain. Application has client side - where people can apply the job. And the admin panel where all these applications and contacts are stored. And other very important nginx role - is to serve gzip files. All js libraries weights about 700Kb and about the same is css libraries. So after compression it weights about 220Kb. 3X compression, not bad, ha!


Back-end architecture is lifted on VPS provided by Vscale.io. Since we dont't really know business model, it will be handy to have a script which deploys blank application on new server. So I decided to play with Ansible which is considered to be one of the best automation tools of nowadays.
